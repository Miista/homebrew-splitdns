# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`hemma` is a Go CLI that generates split-horizon DNS records (Pi-hole/dnsmasq `address=` directives)
and Caddy site blocks for a homelab, from a single declarative `services.yaml` committed to the homelab
git repo. It is **reconcile-and-report** (Terraform/`make` style): every sync re-derives all output from
the YAML. It **does not** deploy, reload services, or SSH anywhere ‚ÄĒ it only writes files into the local
repo checkout. `design.md` is the authoritative design document; read it before non-trivial changes.
Section references in code comments (`design ¬ß6`, etc.) point at it.

The tool was named `sd`, then `splitdns`, and is now `hemma` (Swedish "at home") ‚ÄĒ it outgrew
"split DNS" once it also managed Caddy and auth-provider config. Released artifacts ship a
`splitdns` alias (brew/deb symlink), and both legacy generations of on-disk names (manifest,
generated filenames, Caddyfile import line, .gitignore markers) are auto-migrated ‚ÄĒ see
`legacyManifestNames`, `legacyImportLines`, and `legacyGiBlockStarts` in `internal/cli`.

## Build / test / release

```sh
go build ./...                       # build
go test ./...                        # all tests
go test ./internal/plan/             # one package
go test ./internal/cli/ -run TestFoo # one test
go run ./tools/genman                # regenerate man/hemma.1.gz from CLI help
./deploy.sh [host ...]               # cross-compile + scp to hosts (default: optiplex pi)
```

`deploy.sh` stamps the version via `-ldflags "-X hemma/internal/cli.Version=$(git describe)"`; the
`FLEET` var (`host:GOARCH` pairs) maps hosts to arches. The script is kept portable to bash 3.2 (macOS).

Per the user's global convention: build and live-verify on the Mac before committing/pushing in this repo.

## Architecture ‚ÄĒ the sync pipeline

Data flows one direction through five packages. The layering is deliberate; respect it.

```
services.yaml ‚Üí config ‚Üí plan ‚Üí render        (desired state)
                          ‚Üď
                  sync (Engine) ‚Üź‚Üí manifest    (reconcile + safe GC)
                          ‚ÜĎ
                        cli                    (parse, mutate YAML, invoke sync)
```

- **`internal/config`** ‚ÄĒ loads/mutates/persists `services.yaml`, the sole source of truth. The tool
  **owns** this file and rewrites it wholesale; ordering and human comments are not preserved. Output-path
  segments live here as data (per-host overridable), not hardcoded.
- **`internal/plan`** ‚ÄĒ validates each service **independently**, collecting per-entry errors instead of
  stopping (partial success is the norm). Produces the desired set of `(path, content)` files. Also emits
  synthetic `@domain:` TLS-snippet owners ‚ÄĒ filter with `plan.IsSyntheticOwner`.
- **`internal/render`** ‚ÄĒ pure string generation for the two output types. All generated files carry the
  `# GENERATED by hemma ‚ÄĒ do not edit` header. Caddy ordering matters: TLS snippets must be defined
  before the site blocks that import them.
- **`internal/sync`** ‚ÄĒ the **Engine**: the ONLY code that writes or deletes generated files (single-writer
  invariant, design ¬ß6). `Incremental` mode writes/updates but never deletes; `Complete` mode also GCs
  orphaned manifest-tracked files. Diffs the plan against the manifest.
- **`internal/manifest`** ‚ÄĒ records which files each service generated (`hemma-manifest.yaml`). It is the
  **authority for safe deletion**: only manifest-tracked files are ever deleted. Unparseable ‚Üí rebuilt from
  config. (`splitdns-manifest.yaml` and `sd-manifest.yaml` are the pre-rename names, auto-migrated
  on first load.)
- **`internal/cli`** ‚ÄĒ command parsing and wiring; a thin `main.go` just calls `cli.Run`. add/update/remove
  mutate the YAML then call the shared sync engine ‚ÄĒ they contain **no file-writing logic of their own**.

### Invariants to preserve when editing

- **Single writer**: only `internal/sync` writes/deletes generated files. Never write generated output from
  `cli` or elsewhere.
- **Validate before persist**: mutation commands (`cmdAdd` etc.) validate required flags, fqdn-matches-a-domain,
  and host-exists **before** touching the YAML, so a mistyped command never persists a half-formed entry.
- **`runSync` is the single sync path**, invoked as the tail of every mutation. Mode is chosen by mutation
  shape: add/update/enable/disable use `Incremental` (can't orphan); remove-service and every host/domain/dns-host
  mutation use `Complete` (they can orphan files that must be GC'd, else `apply` refuses on drift).
- **Report-but-proceed**: per-entry validation failures are skipped and reported; the only globally fatal
  condition is YAML that won't parse. Exit non-zero if any entry was skipped (design ¬ß8).

## CLI conventions

- **Verb-first, `<verb> <noun> <args>`** across the whole CLI (`add service`, `remove host`) ‚ÄĒ one word order
  avoids add-service/service-add confusion. Routed via `dispatchNoun` in `cli.go`.
- Operates on `~/docker` by default; `-C`/`--chdir <dir>` overrides (git-style).
- Long + short flag aliases (`--fqdn`/`-f`); `version` verb; `--help` and `help <cmd>` per command.
- **`UsageText` in `cli.go` is the single source of truth for help** ‚ÄĒ `tools/genman` compiles the same
  strings into the man page, so `--help` and the man page can never drift. Edit help text there.
- Color/glyphs (`‚úď ‚úó ‚ö†`) are gated on stdout being a TTY and `NO_COLOR` unset, so piped/captured output stays plain.

## Host-split commands (`apply`, `verify`)

`apply` and `verify` are **host-split**: they identify which host they run on via a local-IP match and
perform only the half they own. The DNS half (restart pihole / dig against it) runs only on the resolver;
the Caddy half (validate + reload) only on a host running Caddy. Neither SSHes elsewhere ‚ÄĒ run on each host.
`apply` runs `caddy validate` **before** `caddy reload` so a missing/wrong cert aborts with a clear error
instead of failing mid-reload; it refuses if the repo has drift.

## Scope boundary (do not cross)

hemma manages the **internal horizon only** (Pi-hole record + Caddy block). It deliberately does not
create, verify, or warn about the **public horizon** (public DNS + tunnel ingress) ‚ÄĒ on this homelab that is
a `cloudflare.io/hostname` label read by cloudflared-wrapper off the docker socket, and it lives in a
hand-maintained `docker-compose.yml` hemma does not own. See `design.md` ¬ß12 for the full rationale.
A service can have a correct internal horizon yet be publicly broken because that label is missing.
